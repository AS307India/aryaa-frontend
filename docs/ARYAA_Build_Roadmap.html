How to Use This Document

One Unit = One Cursor Session

Every unit in this document is a self-contained coding session designed to be completed in 2–4 hours using Cursor Pro with Claude. Each has a ready-to-paste prompt with full context, file paths, dependencies, and a test to confirm it works before you move on.

01

Pick a Unit

Open a phase, find the next unit in sequence. Units are ordered — each one builds on the previous. Never skip ahead.

02

Paste the Prompt

Click the unit to expand it. Copy the full prompt and paste it into Cursor. The prompt contains all context Cursor needs.

03

Run the Test

Every unit ends with a Test step. Run it before moving on. If it fails, fix it before starting the next unit — do not skip.

04

Ask for More

Need the next phase's prompts? Ask Claude: "Give me the prompts for Phase 1, Week 3." You'll get the full set instantly.

⚠️
Critical rule: Never start a unit without completing and testing the previous one. These units have dependencies. Unit 1-12 (SMS dispatch) requires Unit 1-11 (SOS trigger endpoint) to exist. Skipping ahead creates a broken codebase that is very hard to untangle. One unit at a time, always tested.
The Prompt Format

What Every Prompt Looks Like

Every unit prompt follows this exact structure. Cursor needs all of these fields to produce clean, correctly-placed code.

PROMPT TEMPLATE ═══════════════════════════════════════════════════════ UNIT [ID] — [TITLE] ═══════════════════════════════════════════════════════ CONTEXT: What exists in the codebase right now at this point. What the previous unit built. What this unit connects to. STACK: Exact tool + version being used for this unit. TASK: Precise description of what to build. Single focused scope. No ambiguity about what "done" looks like. INPUT: What the function / endpoint / component receives. OUTPUT: What it returns, renders, or does when complete. FILE LOCATIONS: Exact paths where new files are created or existing files are edited. Example: backend/src/routes/auth.ts (create) backend/src/services/sms.ts (edit, add function) DO NOT: Explicit list of things to avoid — scope boundaries, patterns to not use, things the next unit handles instead. DEPENDENCIES: npm install commands to run before starting, if any. TEST: Exact steps to verify this unit works before moving to the next. Must be runnable — a curl command, a screen tap, a log output.
💡
Why this format works: Cursor's biggest weakness is scope creep — it tries to build everything at once and produces broken, tangled code. The DO NOT field and single-scope TASK field prevent this. The TEST field means you never move forward with broken code. The FILE LOCATIONS field means Cursor puts code in the right place instead of inventing its own structure.
