Identified Problems & Setbacks

Every critical issue we identified before writing a single line of code

🔴 Critical Technical Issues
Problem
App Gets Killed in Background
Android battery optimization and iOS background app policies kill inactive apps. SOS may silently fail when the screen is off or the app isn't in focus — the most dangerous failure mode.
Solution
Foreground Service + Heartbeat
Android Foreground Service with persistent notification. iOS Background App Refresh + silent push wakeup. WorkManager heartbeat every 15 mins. Daemon process independent of UI.
Problem
Voice Trigger on Lock Screen
iOS completely blocks third-party microphone access on the lock screen. Android allows it but destroys battery life with constant hotword detection.
Solution
Physical Button Combo Instead
Volume Down + Power × 3 works on lock screen without microphone permission. On Android, hotword detection only activates while charging. iOS gets a lock screen widget instead.
Problem
SMS Delivery Not Guaranteed
Indian carrier SMS can take 30–120 seconds or fail silently. No delivery confirmation by default. Single gateway = single point of failure.
Solution
Dual Gateway + DLR Escalation
Twilio + MSG91 as parallel SMS gateways. Delivery Receipt (DLR) monitoring. If delivery fails in 30s, auto-escalate to a phone call. Pre-composed SMS stored locally for zero processing delay.
Problem
Recording Lost on No-Internet
Secret recording is useless if the phone has no connectivity. If the phone is damaged or snatched, a single large video file may never upload.
Solution
Chunked Encrypted Local-First Upload
Encrypt and store locally first (AES-256). Upload in 30-second chunks. Dead man's switch: if offline 10+ mins during SOS, push last known location + cached chunks to contacts.
Problem
Safe Route Data Doesn't Exist
Reliable real-time data on street lighting, police presence, and open businesses for Pune's roads simply does not exist in any API today.
Solution
Layer Own Data on MapMyIndia
Phase 1: MapMyIndia routing avoiding isolated roads via traffic density. Phase 2: Community map data as avoidance zones. Phase 3: Partner with PMC + Pune Police for official CCTV/lighting data.
🟠 Legal & Compliance Issues
Problem
Secret Recording Legality
Under IT Act + IPC, recording someone without consent in private spaces can be illegal. The platform facilitating such recording could be held liable.
Solution
Two-Layer Consent Architecture
Users consent at onboarding (fingerprint/signature). Emergency contacts consent when joining Trusted Circle. Recording is self-protection, not interception — legally defensible. Get formal legal opinion (₹15k–30k).
Problem
DPDP Act 2023 Compliance
India's Digital Personal Data Protection Act 2023 requires explicit consent per data type, verifiable parental consent for minors, data minimisation, and a Data Protection Officer.
Solution
Privacy Dashboard + DPO Appointment
Granular consent per data type. Privacy Dashboard where users see and delete all their data. OTP-based parental consent for Child Mode. India-based servers (AWS Mumbai). Appoint a DPO from Day 1.
Problem
Community Map Defamation Risk
Anonymous reports pinning specific locations as unsafe can lead to defamation suits from business owners or residents in those areas.
Solution
Aggregate-Only + Infrastructure Reports
Minimum 3 independent reports before a pin appears. Report categories are infrastructure-only ("poorly lit," "no CCTV"). 30-day auto-expiry. IT Act Section 79 safe harbour via grievance redressal mechanism.
Problem
Trusted Circle Can Enable Stalking
A controlling partner or abusive family member could weaponize live tracking to monitor a domestic abuse survivor — the exact person the app is meant to protect.
Solution
Privacy Pause + Secret Exit Feature
Privacy Pause: pause location sharing for 24h without notification. Secret Exit: hidden gesture removes all circle members instantly. Monthly circle reminder with one-tap removal. Co-designed with Pune-based NGOs.

Technical Requirements Document (TRD)

How the product is built — for developers and architects

TRD-01 Architecture Overview

+
Architecture Pattern
API-First Microservices. All features exposed as REST APIs consumed by mobile apps. Enables future web dashboard, smartwatch, and government integrations without rebuilding logic.
Mobile Framework
React Native (Expo) — single codebase for iOS and Android. Reduces development cost by ~40%. Platform-specific native modules written in Swift (iOS) and Kotlin (Android) only where absolutely required (background foreground service, deep system permissions).
Backend Framework
Node.js with Express / Fastify. PostgreSQL as primary database. Redis for session management, queuing, and rate limiting. All hosted on AWS Mumbai (ap-south-1) region for DPDP Act compliance.
Multi-City Database Design
All records include city_id field from Day 1. Adding Mumbai = adding data, not rewriting schema. Geographic data partitioned by city for performance.
Message Queue
AWS SQS for SOS notification dispatch. Handles burst load (1000 simultaneous SOS triggers) without server overload. SMS, push, and call dispatch are separate queue workers.

TRD-02 Background Process Architecture

+
Android Implementation
  • Android Foreground Service: persistent notification "ARYAA is protecting you" keeps process alive
  • Device Admin App registration: prevents easy uninstall without user action
  • WorkManager heartbeat: 15-minute health ping to backend confirms app is alive
  • Onboarding flow: step-by-step guide to whitelist ARYAA from battery optimization (per manufacturer: Samsung, Xiaomi, OnePlus, etc.)
  • Sticky broadcast receiver: re-launches daemon if killed by OS
iOS Implementation
  • Background App Refresh enabled with regular significant location change monitoring
  • Silent push notifications from backend wake app every 15 minutes
  • CoreLocation significant location change API: wakes app even when killed
  • Lock screen widget (iOS 16+) for one-tap SOS without unlocking
  • Siri Shortcut integration: "Hey Siri, ARYAA SOS" as voice trigger fallback

TRD-03 SOS Flow — Detailed Technical Sequence

+
T=0ms — Trigger
User taps SOS button / hardware button combo. Local daemon receives intent. App foregrounds if backgrounded.
T=200ms — Local Processing
Read emergency contacts from local SQLite. Get last cached GPS (if live GPS not available in 500ms). Compose pre-formatted SMS strings from templates stored locally.
T=500ms — SMS Dispatch
Android: SmsManager sends SMS directly from device (no internet required). iOS: API call to Twilio/MSG91 (requires internet). If no internet on iOS: queue SMS for delivery on reconnect, send iMessage if available.
T=800ms — Push + API Call
If internet available: POST /api/sos to backend with user ID, GPS coordinates, timestamp. Backend dispatches push notifications to all contacts' devices. Begins live tracking session.
T=1500ms — Recording Start
Camera/microphone recording begins in background. First 30-second chunk queued for encrypted upload.
T=3000ms — 112 Countdown
5-second countdown timer starts for auto-call to 112. User can tap "Cancel auto-call" to prevent it. If no cancel: Twilio Programmable Voice initiates call to 112 via backend.
Delivery Receipt Monitoring
Backend checks SMS DLR every 10 seconds. If any contact hasn't received SMS within 30 seconds: auto-retry via backup gateway. If retry fails: initiate automated voice call to that contact.

TRD-04 Security Architecture

+
Data Encryption
  • All data in transit: TLS 1.3 (no fallback to 1.2)
  • All data at rest: AES-256-GCM
  • Phone numbers stored as HMAC-SHA256 hashes + encrypted — never plaintext
  • Recordings: end-to-end encrypted — server cannot decrypt, only the user's key can
  • GPS history: encrypted with user-derived key, deleted per user retention setting
API Security
  • JWT tokens with 24-hour expiry + refresh token rotation
  • Rate limiting: SOS endpoint — 10 calls/minute per user (prevents spam)
  • Certificate pinning on mobile apps: prevents MITM attacks
  • All API endpoints: input validation, SQL parameterisation, no raw queries
Infrastructure Security
  • AWS VPC with private subnets — database not publicly accessible
  • WAF (Web Application Firewall) on all public endpoints
  • CloudTrail audit logging: every data access logged with timestamp and user
  • Automated vulnerability scanning via AWS Inspector

Legal & Compliance Document

Every legal obligation and how ARYAA meets it

LEGAL-01 DPDP Act 2023 Compliance Checklist

+
Consent Requirements
  • ✅ Separate, granular consent for each data type: location, contacts, microphone, camera, recordings
  • ✅ Consent withdrawal mechanism available at any time in Privacy Dashboard
  • ✅ OTP-based verifiable parental consent for Child Mode (under-18 users)
  • ✅ Emergency contacts must actively accept consent notice before joining Trusted Circle
Data Minimisation & Retention
  • ✅ Location history: user-configurable retention (7 / 30 / 90 days), auto-delete after
  • ✅ Recordings: auto-delete after 30 days unless user saves explicitly
  • ✅ No data collected or processed beyond what's listed in consent
  • ✅ Children's data: stricter 7-day retention maximum, no analytics use
Data Localisation
All Indian user data hosted exclusively on AWS Mumbai (ap-south-1). No data transferred to foreign servers. CloudFront CDN configured with India edge nodes only for Indian users.
Data Protection Officer
DPO appointed from founding team. DPO contact published in app's Privacy Policy and accessible from main menu. DPO reachable within 72 hours for data-related grievances.

LEGAL-02 Recording Consent Framework

+
User Consent (Onboarding)
Text shown at onboarding (requires active acknowledgement): "When you activate SOS, ARYAA will automatically begin recording audio and video. These recordings are end-to-end encrypted and stored securely. You consent to this recording being shared with emergency contacts and law enforcement when SOS is active. You can disable this feature at any time in Settings."
Legal Basis
User is recording their own immediate environment for self-protection. Indian courts have consistently upheld self-protection recordings. IT Act privacy provisions apply to interception of communications — a person recording their own surroundings is not interception. Formal legal opinion required from qualified advocate before App Store launch (budget: ₹15,000–30,000).
IT Act Section 79 Safe Harbour
ARYAA qualifies for safe harbour under Section 79 of the IT Act as an intermediary provided: (1) grievance redressal mechanism exists, (2) user-generated content is moderated per government guidelines, (3) platform does not initiate transmission. All three conditions are built into ARYAA's design.

Phase 5 — Universal Smartwatch Layer

Any watch. Any budget. The same SOS power on your wrist.

💡
The Core Insight: Budget Indian smartwatches (boAt, Noise, Fire-Boltt) cannot run third-party apps — they run locked proprietary firmware. So ARYAA doesn't run on the watch. The phone does all the work. The watch is just the trigger. This is the only approach that works universally across every smartwatch at every price point.
Why Wrist-Based SOS Is a Genuine Upgrade
📵 Phone Is Unreachable

In most real emergencies, the phone is in a bag, a pocket, or has been grabbed. The watch is always on the wrist — always accessible, always accessible even with one hand occupied.

👁️ Zero Visual Tell

Tapping your watch looks like checking the time. There is no visible signal to an aggressor that you are calling for help. This is meaningfully more discreet than unlocking a phone.

🧓 Senior Citizen Game-Changer

A senior who falls and cannot reach their phone can trigger SOS from their wrist. Combined with fall detection via accelerometer, this is a genuinely life-saving capability for this demographic.

Three-Tier Integration Strategy
Tier 1 BLE Bridge — Budget Watches (boAt, Noise, Fire-Boltt, Fastrack, Titan Smart) ₹1,500–₹5,000 watches

These watches cannot run apps. But every one of them communicates with their companion phone app via Bluetooth Low Energy (BLE). ARYAA's phone app runs a background BLE listener that watches for a specific signal pattern.

How it works technically: When the user presses a button combo on their watch (e.g. hold side button 3 seconds), the watch sends a BLE notification to the phone — normally this would open the watch's own companion app. ARYAA intercepts this BLE event and triggers SOS instead. No firmware modification. No jailbreak. Just BLE event listening.

  • Works on any BLE-enabled watch regardless of brand or price
  • User pairs their watch to ARYAA once during setup — ARYAA learns the watch's BLE device ID
  • Button combo is user-configurable to avoid accidental triggers
  • Vibration feedback from watch confirms SOS was received by phone
  • Battery impact on phone: minimal — BLE listening uses ~1–2% extra battery per day
  • Works even if watch companion app is not installed
Tier 2 Wear OS Native App — Mid-Range Android Watches ₹5,000–₹20,000 watches

Watches running Wear OS (Samsung Galaxy Watch, Fossil, TicWatch, Mobvoi) can install third-party apps directly. This enables a proper ARYAA watch app with its own UI on the watch face.

  • Dedicated ARYAA watch face tile with one-tap SOS button
  • Displays active SOS status and countdown on watch screen
  • Heart rate spike detection — if BPM suddenly jumps above threshold, prompt user "Are you safe?"
  • Real-time vibration alerts when a Trusted Circle member triggers their SOS
  • Shows nearest police station distance on watch face during SOS
  • Works independently of phone for SOS trigger (sends via paired phone's connection)
  • Fall detection using Wear OS accelerometer API
Tier 3 Apple Watch (watchOS) Native App ₹25,000+ watches

Apple Watch has the most powerful safety APIs available — including hardware fall detection, ECG, crash detection, and Emergency SOS built into the OS. ARYAA's watchOS app layers on top of these.

  • Native watchOS SwiftUI app — separate Swift codebase required
  • Complication on watch face: pulsing ARYAA SOS dot always visible
  • Hardware fall detection API: if Apple Watch detects a fall and user doesn't respond in 30s, ARYAA triggers SOS automatically
  • ECG/heart rate anomaly hook: extreme BPM triggers safety check prompt
  • Haptic SOS confirmation: specific tap pattern tells user SOS was sent
  • Digital Crown + Side Button combo triggers ARYAA SOS (distinct from Apple's own SOS)
  • Watch operates independently — can trigger SOS even if phone is out of range (via LTE models)

Security Architecture

Every threat ARYAA faces, in detail — and exactly how we defend against each one

⚠️
Why security is non-negotiable for ARYAA specifically: Most apps losing data means embarrassment. ARYAA losing data means live locations of abuse survivors exposed, emergency contacts compromised, SOS silenced during real danger. Every security decision must be made before code is written — retrofitting security is 10× harder and 100× more expensive than building it in from the start.
🔴 Category 1 — Attacks That Can Get People Killed
Threat 1 — Problem
SOS Spoofing & API Flooding
An attacker discovers your SOS endpoint and floods it with thousands of fake triggers from fake accounts. Your SMS gateways hit their limits, your server crashes, and real users' SOS messages never go out. MSG91 and Twilio both charge per SMS — a spoofing attack can also run up a ₹50,000+ gateway bill in minutes, making it a financial weapon too. Unlike a normal DDoS this attack specifically silences real emergencies.
Solution
Multi-Layer SOS Rate Limiting + Billing Caps
Three independent rate limit layers: (1) Per user: max 10 SOS triggers per hour per authenticated account. (2) Per IP: max 50 requests per minute across all endpoints, regardless of account. (3) Per gateway: hard daily SMS spend cap of ₹2,000 — if breached, pause and alert you instantly. Additionally, SOS endpoint requires valid JWT + matching device fingerprint — a stolen token from a different device is rejected. AWS WAF blocks known bot signatures before they reach your server.
Threat 2 — Problem
Live Location Data Interception
Your app continuously transmits GPS coordinates to your server. On a compromised WiFi network, a rogue cell tower (IMSI catcher), or via a man-in-the-middle attack, these transmissions can be intercepted and read in real time. For a domestic abuse survivor using ARYAA, her live location being silently tracked by her abuser is the exact threat she is trying to escape. ARYAA would become the weapon used against her.
Solution
Certificate Pinning + Payload Encryption + No Location in Push
TLS 1.3 for all data in transit — no fallback to older versions. Certificate pinning in the mobile app: the app only communicates with your specific server certificate; even on a compromised network, MITM fails because the certificate won't match. Never send raw coordinates in push notification payloads — send only a session token; the app fetches location from the server using that token. Intercepted push notification = useless token. Location data encrypted with a per-user AES-256 key, not a single master key — one compromised user affects zero others.
Threat 3 — Problem
Emergency Contact Hijacking
If account takeover protections are weak, an attacker who gains access to a user's account can silently replace their emergency contacts with numbers they control. The user believes they are alerting family during an emergency. They are alerting the attacker. This is particularly dangerous in domestic abuse scenarios where the abuser may have access to the victim's phone number for SIM swap purposes.
Solution
Contact Change Re-Authentication + Audit Trail
Any change to emergency contacts requires re-authentication: the user must enter their PIN or biometric, plus receive a confirmation OTP on the registered number. Contact changes are logged with timestamp, IP, and device fingerprint — visible to the user in their Privacy Dashboard. Changing all contacts simultaneously triggers a security flag and a "Did you make this change?" notification to the device. Previous contacts receive an SMS: "You have been removed from [user]'s ARYAA safety circle."
Threat 4 — Problem
Fake SOS Cancellation Attack
During an active SOS, if the cancel endpoint is not properly authenticated and re-verified, an attacker with a valid stolen token can send a cancellation signal — making all emergency contacts believe the user is safe when they are not. The contacts stand down. The real emergency continues with no help coming.
Solution
Biometric / PIN Re-Auth to Cancel + Duress PIN
Cancelling an active SOS requires the user to enter their PIN or biometric on-device — not just a valid JWT. This means an attacker with a stolen API token cannot cancel remotely. Additionally, implement a Duress PIN (see Recommended Additions): a second PIN that appears to cancel SOS but actually continues it silently and sends a "user is under duress" flag to contacts. Modelled on bank ATM duress PIN systems.
🟠 Category 2 — Infrastructure Attacks
Threat 5 — Problem
DDoS — Distributed Denial of Service
A coordinated flood of traffic from thousands of IP addresses overwhelms your server, taking everything offline — SOS, tracking, all of it. Unlike a shopping app where downtime loses sales, ARYAA downtime during peak hours (10 PM–2 AM when most safety incidents happen) directly endangers users whose SOS never fires. Attackers don't need a specific motive — DDoS-for-hire services cost as little as $5/hour on dark web marketplaces.
Solution
CloudFront + AWS Shield + Auto-Scaling
CloudFront CDN sits in front of your entire API — it absorbs layer 3/4 volumetric DDoS automatically and has geographic rate limiting and bot detection built in. AWS Shield Standard is free and already active on all AWS accounts, handling common network-level attacks. Auto-scaling groups ensure your EC2 fleet scales horizontally under load — traffic spikes spin up new instances automatically. For Phase 2+: AWS Shield Advanced (₹18,000/month) adds 24/7 DRT (DDoS Response Team) support and financial protection against DDoS-caused billing spikes.
Threat 6 — Problem
API Enumeration & Endpoint Scraping
Attackers systematically probe your API — trying every endpoint, every parameter combination — to map your entire backend structure. Once they understand your API, every subsequent attack becomes easier. They can harvest user IDs, discover undocumented or debug endpoints left in production, find parameters that bypass auth checks, or identify which endpoints have weaker rate limiting to use as vectors.
Solution
WAF + API Gateway + Opaque IDs + Honeypot Endpoints
AWS WAF blocks known enumeration patterns and sequential request probing. All user-facing IDs are UUIDs (random 128-bit), never sequential integers — you cannot guess user ID 1001 by incrementing from 1000. All API errors return generic 404 or 400 responses — never reveal whether an endpoint exists or what parameters it expects. Add 3–5 honeypot endpoints that look real but trigger an immediate IP block and security alert when accessed. Any legitimate user would never hit these — only scanners do.
Threat 7 — Problem
SQL Injection
If any API endpoint takes user input and constructs a database query by string concatenation without sanitisation, an attacker can inject malicious SQL to extract the entire database, delete records, bypass authentication, or escalate to server-level access. Classic attack — discovered in 1998. Still devastatingly effective in 2025 because developers moving fast skip parameterisation. One vulnerable endpoint is enough to expose every user's data.
Solution
Parameterised Queries + ORM + WAF SQLi Rules
Absolute rule: zero raw SQL string construction anywhere in the codebase. Use an ORM (Prisma or Drizzle for Node.js) which parameterises all queries by default — the ORM makes SQLi structurally impossible for standard queries. For any raw queries that must be written, use prepared statements with bound parameters exclusively. AWS WAF has a managed SQLi rule group that blocks known injection patterns at the network layer as a second line of defence. Code review checklist includes explicit SQLi check on every PR that touches database code.
Threat 8 — Problem
Server-Side Request Forgery (SSRF)
If your server fetches external URLs based on user input — for example, processing a user-submitted link, fetching a map tile, or following a redirect — an attacker can point that request to your internal AWS infrastructure. The most dangerous target: the AWS instance metadata endpoint at 169.254.169.254, which returns the EC2 instance's IAM credentials in plaintext. An attacker with those credentials has full access to your AWS account — S3 buckets, databases, everything.
Solution
IMDSv2 + URL Allowlist + No User-Controlled Fetches
Enforce IMDSv2 on all EC2 instances — this requires a session token for metadata access, making simple SSRF attacks against 169.254.169.254 fail. Architectural rule: the backend server never fetches arbitrary URLs provided by users. Any external URL fetching uses a strict allowlist (only MapMyIndia, MSG91, Twilio, FCM domains). Outbound requests from the server go through a dedicated egress proxy that enforces the allowlist at the network level — not just in code.
🔵 Category 3 — Mobile & Client-Side Attacks
Threat 9 — Problem
APK Reverse Engineering & Hardcoded Secrets
On Android, anyone can download your APK and decompile it in minutes using tools like jadx or apktool — both free and widely used. If API keys, server URLs, encryption keys, or secret tokens are hardcoded in the app code, they are fully exposed. An attacker can then call your APIs directly using those keys, bypassing authentication entirely, or use your MSG91/Twilio keys to send SMS at your expense. This is the most common mistake made by Indian developers building their first app.
Solution
Zero Secrets in Binary + Code Obfuscation + Remote Config
Absolute rule: nothing sensitive in the app binary. No API keys, no server URLs, no tokens, no encryption keys — ever. All configuration fetched from a secure remote config endpoint at startup, authenticated by a rotating app-level token. ProGuard + R8 on Android for code obfuscation — makes decompiled code significantly harder to read and reverse engineer. iOS: Swift obfuscation via LLVM passes. The compiled APK/IPA should be fully readable by anyone and contain nothing sensitive. Regularly audit with tools like MobSF (Mobile Security Framework) to verify no secrets leaked into builds.
Threat 10 — Problem
Certificate Pinning Bypass
Attackers use tools like Frida, SSL Kill Switch 2, or objection on rooted/jailbroken devices to bypass TLS and intercept app traffic even when HTTPS is in use. They install a custom certificate on the device and your "secure" connection becomes readable plaintext. This exposes live location data, SOS triggers, and contact information in transit. Tools to do this are freely available and require only basic technical knowledge.
Solution
Root Detection + Anti-Tampering + Certificate Pinning
Certificate pinning: the app hardcodes your server's public key hash and refuses connections to any server with a different certificate — even one with a valid CA signature. Root/jailbreak detection using RootBeer (Android) and DTTJailbreakDetection (iOS) — if the device is rooted, disable sensitive features and warn the user. Anti-tampering checks: the app verifies its own signature on startup — if it has been repackaged (common with pirated APKs), it exits. These layers together mean that even on a compromised device, attackers face multiple independent barriers.
Threat 11 — Problem
BLE Spoofing (Phase 5 Specific)
When the BLE bridge is built for budget smartwatches, an attacker with a BLE transmitter (available for ₹500 on Amazon) can spoof a watch's BLE signal and trigger SOS on any ARYAA user's phone within Bluetooth range — or trigger false cancellations during real emergencies. In a crowded environment like a college campus or IT park, a single attacker could trigger mass false SOS alerts, flooding emergency contacts and desensitising them to future real alerts.
Solution
Rolling HMAC + Proximity Check + BLE Rate Limiting
For Wear OS and Apple Watch (Tier 2/3): BLE trigger includes a rolling HMAC code that changes every 30 seconds (TOTP-style). A replayed or spoofed BLE signal has an expired HMAC and is rejected. For budget watches (Tier 1) that can't generate HMAC: minimum hold duration of 2+ seconds (attacker would need to hold signal for 2s — difficult to spoof convincingly), plus RSSI proximity check — signal strength must indicate the device is within 1 metre of the phone. BLE trigger rate limiting: maximum 3 SOS triggers per 5 minutes from any BLE source. Watch-to-phone pairing uses a shared secret generated at pairing time — only the paired device can trigger SOS.
Threat 12 — Problem
Malicious Trusted Circle Member
Not a technical hack — a social engineering attack. An abusive partner or controlling family member persuades the victim to add them to the Trusted Circle, giving them full legitimate access to live location data. The app functions exactly as designed. The outcome is dangerous. This is not a hypothetical — domestic abuse situations frequently involve digital surveillance disguised as "care." This is the hardest threat to defend against technically because the attacker is an authorised user.
Solution
Privacy Pause + Secret Exit + Relationship Safety Screening
Privacy Pause: pause location sharing with any specific circle member for up to 24 hours without notification to them — no explanation required. Secret Exit: hidden gesture (tap logo 5×) instantly removes all circle members and disables all location sharing — designed specifically for this scenario. Monthly circle reminder: every member of the Trusted Circle receives a monthly notification that they have access, with a one-tap removal option. Onboarding relationship safety screen: "Is there anyone in your life who makes you feel unsafe? You can exclude them from all access." Importantly, users are never required to justify removing someone from their circle — the app never asks "are you sure?" in a way that creates friction for leaving.
Threat 13 — Problem
Push Notification Payload Exposure
If push notification payloads contain sensitive data — GPS coordinates, SOS status, contact names — in the notification body, that data is exposed to multiple risk surfaces: malicious apps on the same device that intercept notifications, notification previews on lock screens visible to anyone, notification logs stored on device that other apps can read with the right permissions, and FCM/APNs logs on Google/Apple infrastructure. For a user in danger, a notification preview saying "SOS triggered at [exact address]" visible on a lock screen is a serious privacy failure.
Solution
Token-Only Push Payloads + Silent Notifications
Push notification payloads contain only: a session token, an event type (e.g. "SOS_UPDATE"), and a timestamp. No location data, no names, no status details. The app receives the silent push, wakes up in background, authenticates with the server using the token, and fetches the actual data over encrypted HTTPS. Push notification body visible on lock screen: generic text only — "ARYAA Safety Alert" — never specific details. Use silent/data-only push notifications for all SOS-related events so they never appear as visible notifications on the recipient's device at all — the app handles display itself.
🟢 Category 4 — Data & Account Attacks
Threat 14 — Problem
OTP Bypass & Brute Force
Your entire authentication chain relies on OTP delivered to a phone number. Standard 6-digit OTPs have only 1,000,000 possible values. Without rate limiting and lockout mechanisms, an attacker can brute force a valid OTP — trying all combinations — in minutes. If the OTP window is 10 minutes, an attacker needs only to send ~600 guesses per minute to have a good chance of success. This gives them full account access: location history, emergency contacts, recordings.
Solution
Rate Limiting + Progressive Lockout + Short Windows
OTP valid window: 3 minutes maximum (not 10). Maximum 3 OTP attempts per session — after 3 failures, the OTP is invalidated and a new one must be requested. Maximum 5 OTP requests per phone number per hour — after this, a 1-hour cooldown. Progressive lockout: after 3 failed sessions in 24 hours, account is locked and requires a manual unlock via a secondary verification channel. OTPs are single-use and server-side invalidated immediately after one successful use — cannot be reused even within the valid window. All OTP events logged with IP, device fingerprint, and timestamp for anomaly detection.
Threat 15 — Problem
SIM Swap Attack
A SIM swap attack is when an attacker calls the victim's mobile carrier, impersonates them using publicly available personal information, and convinces the carrier to transfer the victim's phone number to a SIM the attacker controls. Once successful, the attacker receives all OTPs sent to that number — giving them full control of the ARYAA account. SIM swap is alarmingly easy to execute against Indian carriers and is the primary method used to compromise OTP-based accounts. The attacker then has access to live location, contact list, and recording history.
Solution
Device Binding + SIM Change Detection + Secondary Verification
Device fingerprint binding: after first login, the account is bound to that device's hardware fingerprint (Android ID, iOS identifierForVendor). Login from a new device requires: (1) OTP to the phone number AND (2) confirmation from the original device if available. SIM change detection: ARYAA checks the device's SIM serial (ICCID) on each app launch. If the ICCID changes (indicating SIM swap or SIM change), the app immediately locks and requires re-verification before any location data is accessible. Users are notified: "Your SIM card appears to have changed. Please verify your identity." This doesn't prevent the SIM swap but detects it immediately and locks the account before damage is done.
Threat 16 — Problem
Recording Storage Breach (S3 Misconfiguration)
Your S3 bucket stores encrypted emergency recordings. S3 misconfiguration — accidentally setting a bucket to public — has exposed data from hundreds of companies including government agencies and Fortune 500 companies. AWS makes it easy to set public access in one click and easy to miss that you've done so. ARYAA's recording bucket being public would expose every encrypted recording to the open internet. If encryption key management is also weak, those recordings become decryptable. Emergency recordings of people in danger are among the most sensitive data imaginable.
Solution
Account-Level Public Access Block + Envelope Encryption + KMS
Enable AWS "Block All Public Access" at the AWS account level — not just the bucket level. This makes it physically impossible for any bucket in your account to ever be made public, even accidentally. Envelope encryption via AWS KMS: each recording is encrypted with a unique data encryption key (DEK). The DEK is encrypted with a master key in AWS KMS. Your code never handles raw keys — KMS does. Even a full database breach gives an attacker only encrypted DEKs, which are useless without KMS access (a separate hardened system requiring separate credentials). End-to-end encryption option: for highest-sensitivity users, recordings are encrypted on-device before upload — your server receives and stores ciphertext it cannot decrypt.
Threat 17 — Problem
Insider Threat
Your own team members or contractors have access to the production database and infrastructure. A rogue employee or a contractor with wider access than needed could export user location histories, contact lists, or recordings. Unlike an external breach, insider threats are hard to detect and often go undiscovered for months. For a safety app whose users include abuse survivors, journalists, activists, and other high-risk individuals, insider access to their location history is potentially life-threatening.
Solution
Principle of Least Privilege + Audit Logging + Access Reviews
Principle of least privilege: every team member and contractor gets database access only to the specific tables their work requires — nothing more. A contractor building the Nearby Help map feature has zero access to the SOS logs table. Access is granted per-task and revoked on task completion. AWS CloudTrail logs every data access with who, what, when, and from which IP — these logs are stored in a separate account that no team member can modify. Quarterly access reviews: every 90 days, review who has access to what and revoke anything no longer needed. All production database queries over a threshold size trigger an automatic alert — bulk exports by anyone are flagged immediately.

Storage Strategy — SQLite to Cloud

How to build the prototype on SQLite and migrate to AWS PostgreSQL at release without rewriting a single line of business logic

💡
The key insight: The database you use doesn't matter nearly as much as how you write the code that talks to it. If you write the data layer correctly from Day 1 using an ORM, swapping SQLite for PostgreSQL at release is a single config line change — not a rewrite. If you write it wrong, migration becomes weeks of painful work.
The Two-Phase Storage Plan
Prototype Phase SQLite — Local File

Your backend runs a local SQLite file on your development machine. Zero infrastructure cost, zero setup time, works completely offline. You test every single SOS flow, tracking feature, contact notification, and auth system — all of it works identically to production. The database just happens to be a file sitting on your laptop.

  • Cost: ₹0
  • Setup time: 10 minutes
  • No internet required for development
  • Instant resets for testing — delete the file, start fresh
  • Full SQL support — every query you write works in PostgreSQL too
Release Phase PostgreSQL on AWS RDS Mumbai

Same backend code. Same Prisma schema. Same queries. One config line changes. Prisma handles the migration automatically. You get production-grade reliability, DPDP-compliant data residency in India, automated daily backups, and the ability to scale horizontally as users grow.

  • Cost: ₹1,200–2,000/month (db.t3.micro)
  • AWS Mumbai region — DPDP compliant
  • Automated daily backups with 7-day retention
  • Encryption at rest enabled in one click
  • Private subnet — never publicly accessible
The Tool That Makes This Work — Prisma ORM
Why Prisma Is the Right Choice for ARYAA

An ORM (Object Relational Mapper) sits between your application code and the database. Instead of writing raw SQL that is specific to SQLite or PostgreSQL, you write database calls in TypeScript using Prisma's syntax — and Prisma translates it to whatever database is configured underneath. Same code. Different database. Zero changes.

What Prisma gives you
  • Define your entire database schema once in a single schema.prisma file
  • Prisma generates fully type-safe database calls — TypeScript catches errors before runtime
  • Switch from SQLite to PostgreSQL by changing one word in the config
  • Migrations are versioned files — every schema change is tracked and replayable
  • Makes SQL injection structurally impossible for standard queries
  • Auto-generates a visual database browser (Prisma Studio) for debugging
The migration in practice
  • Prototype config: provider = "sqlite"
  • Production config: provider = "postgresql"
  • Run prisma migrate deploy on production server
  • Set DATABASE_URL environment variable to RDS connection string
  • That is literally the entire migration if built correctly
The Two Storage Layers — On-Device vs Server-Side
⚠️
Important distinction: ARYAA has two completely separate storage concerns. The on-device SQLite layer never goes away — it is your offline backbone and exists in both prototype and production. The server-side storage is what migrates from SQLite to PostgreSQL. These are independent systems serving different purposes.
Layer 1 — On-Device Storage (Permanent)
SQLite on the User's Phone — Forever
This layer exists in both prototype and production and never migrates anywhere. It is ARYAA's offline backbone — the reason SOS works without internet. Built with WatermelonDB (built on SQLite, designed for React Native, handles sync perfectly).

What it stores:
• Emergency contacts (name, number, relationship)
• Last cached GPS coordinates — rolling 2-hour window, updated every 60 seconds
• Pre-composed SOS SMS templates — ready to send with zero processing time
• Cached nearby locations (police, hospitals, pharmacies) — updated daily when connected
• User preferences and app settings
• Queued outgoing messages when offline — auto-send on reconnect
• Active SOS state — persists even if app is killed and restarted
Layer 2 — Server-Side Storage (Migrates at Release)
SQLite Prototype → PostgreSQL Production
This is what lives on your backend server and migrates from SQLite (prototype) to AWS RDS PostgreSQL (release). Requires internet to access — backed by the on-device layer when offline.

What it stores:
• User accounts and OTP auth tokens
• Live location sessions (Safe Walk, active SOS tracking)
• SOS event logs and audit trails — every trigger, timestamp, GPS, device fingerprint
• Emergency contact relationships between accounts
• Trusted Circle memberships and consent records
• Community safety reports (aggregate, anonymised)
• Recording metadata (S3 key, duration, SOS event reference)
• Binary files (recordings, photos) → stored in AWS S3, never in SQL
5 Rules You Must Follow During Prototype to Avoid Pain Later
01 Use UUIDs as Primary Keys — Never Auto-Increment Integers

SQLite's default primary key is an auto-incrementing integer (1, 2, 3...). PostgreSQL uses UUIDs. If you build your entire prototype with integer IDs, every single foreign key relationship in your database has to be rewritten during migration. This is weeks of work.

The fix from Day 1: In your Prisma schema, every model uses id String @id @default(uuid()). Set it once, never touch it again. UUIDs work identically in SQLite and PostgreSQL.

02 Never Use SQLite-Specific Data Types or Features

SQLite is loosely typed — it will happily store anything in any column. PostgreSQL is strict. If you store phone numbers as integers in SQLite (which loses leading zeros — "091234" becomes 91234), PostgreSQL will reject them on migration. If you use SQLite's AUTOINCREMENT, DATETIME('now'), or any SQLite-only function in raw queries — they break in PostgreSQL.

The fix: Always use Prisma's type system (String, Int, DateTime, Boolean, Json) — Prisma translates these to the correct native type for whatever database is underneath. Store phone numbers as String always. Store dates as DateTime always.

03 Use Prisma Migrations From the Very First Schema Change

Every time you change your database schema — add a column, rename a table, change a type — run it through a Prisma migration file. prisma migrate dev --name describe_your_change. This takes 30 extra seconds per change during development.

Why it matters: This builds your complete migration history. When you provision your PostgreSQL production database on AWS and run prisma migrate deploy, it replays every migration in order and your schema is identical to your local SQLite — every table, every column, every constraint. Without this history, you have to manually recreate the schema and pray you didn't miss anything.

04 Build a Data Access Layer — Never Call Prisma Directly from Route Handlers

Do not write prisma.user.findUnique() directly inside your Express route handler. This couples your business logic to your database library. If you ever need to change the query, add caching, or switch databases, you touch every route file.

The fix: Create a data access layer — a set of plain TypeScript functions: getUserById(), createSOSEvent(), updateLiveLocation(). Your route handlers call these functions. The functions call Prisma. During migration, if anything changes, it changes in one file — not 40 route files.

05 Never Store Binary Files in Any SQL Database

Recordings, photos, audio files, documents — these must never go into SQLite or PostgreSQL. Storing binary files in SQL databases causes serious performance degradation, makes backups enormous, and makes migration nearly impossible. This applies in prototype too — not just production.

The fix: Prototype: store files in a local /uploads folder on your machine. The database stores only the file path as a string. Production: store files in AWS S3. The database stores only the S3 key as a string. The code change from prototype to production is updating one path prefix — the database schema does not change at all.

Three Environments — Build This Separation From Day 1
💻 Local
SQLite

Your daily development environment. SQLite file on your machine. Zero cost. Zero internet needed. Reset instantly by deleting the file. Used by every developer on the team independently.

🧪 Staging
PostgreSQL RDS (small)

A mirror of production on a minimal AWS RDS instance (db.t3.micro — ₹800/month). Test the migration here first. Run your full test suite against real PostgreSQL before anything touches production. Catches type mismatches before real users see them.

🚀 Production
PostgreSQL RDS (proper)

Real users. AWS RDS db.t3.small or above in Mumbai. Automated daily backups. Encryption at rest. Private subnet. Multi-AZ for Phase 2+. Never touched directly — all changes go through staging first.

Recording Feature — Misuse Prevention Framework

How to ensure recordings are captured only in genuine emergencies, and cannot be weaponised for surveillance, revenge, or personal disputes

⚠️
The core tension: The same feature that protects a victim in danger can be weaponised by an abuser for surveillance. The same SOS trigger that captures evidence of an attack can be used to secretly record a private conversation. You cannot tell these two users apart at the moment of recording — both pressed the same button. So your safeguards cannot work at the recording layer. They have to work at the trigger layer, the access layer, and the consequence layer.
The Six-Layer Misuse Prevention Framework
Layer 1 Trigger-Locked Recording — The Most Important Safeguard

Recording must be technically impossible to start without a genuine SOS trigger. This sounds obvious but most apps get it wrong by offering recording as a separate feature.

What Would Go Wrong
Recording as a Standalone Feature
If there is a "Record" button anywhere in the app — even hidden in settings — a malicious user can activate it without triggering SOS. They get a covert recording tool with cloud backup. No alert goes out. No paper trail. No deterrent. The app becomes a spy tool.
The Correct Architecture
Recording as a Side Effect of SOS Only
The recording module is only initialised by the SOS service — not by any user-accessible UI element, API endpoint, or background task. The SOS trigger simultaneously alerts contacts, shares location, AND starts recording — all three are atomic. You cannot have recording without the alert going out first. A malicious user who triggers SOS to record someone immediately sends alerts to all their emergency contacts saying they are in danger. That is a massive deterrent and an instant paper trail. The cost of starting a recording is that your contacts are alerted to a fake emergency — most malicious users will not pay that cost.
Layer 2 Mandatory Alert Coupling — Every Recording Has an Auditable Parent

In your database, a recording cannot exist without a parent SOS event record. The schema enforces this at the database level — not just in application code.

What Would Go Wrong
Orphaned Recordings With No Context
Without schema-level enforcement, a recording could exist in the database with no associated SOS event — either through a bug, a direct database manipulation, or an edge case in the code. This makes it impossible to verify whether any recording was made during a genuine emergency or manufactured maliciously.
The Correct Architecture
Non-Nullable Foreign Key Enforcement
In the Prisma schema: recording.sos_event_id is a non-nullable foreign key — a recording row literally cannot be inserted without a valid SOS event ID. Every SOS event stores: timestamp, GPS coordinates, device fingerprint, user account ID, and trigger method (button tap, hardware combo, watch BLE). For legal proceedings, this audit trail proves the recording was made during a documented emergency — it was not manufactured afterward. If someone claims a recording was made maliciously, you pull the SOS event log and show exactly what happened, when, and where.
Layer 3 Access Controls on Playback — The 24-Hour Lock

Most apps secure recordings in transit and at rest but leave playback completely open. This is where misuse happens — the attacker already has their recording and immediately downloads and shares it.

What Would Go Wrong
Immediate Unrestricted Playback Access
Someone triggers fake SOS to record a private conversation. SOS is immediately cancelled (their contacts were alerted but they make up an excuse). Within 30 seconds they download the recording and share it on WhatsApp. By the time anyone realises what happened, the recording has already spread. The damage is done. The window for intervention was never opened.
The 24-Hour Playback Window
Time-Locked Decryption Key
Recordings are stored encrypted with a key that requires two factors to unlock: the user's own authentication AND a time condition — recordings cannot be accessed for playback by the user themselves for 24 hours after the SOS ends. Why 24 hours? Emergency contacts were already alerted. If the SOS was fake, someone in that person's circle is likely to realise it within 24 hours. This window allows for intervention, reporting, and account suspension before the recording is weaponised. Exceptions to the 24-hour lock: law enforcement with a valid legal request (immediate access), the user's designated legal advocate (immediate access), and a documented ongoing legal case (4-hour manual review by ARYAA team). After 24 hours: legitimate users access their evidence freely for their police report or legal case.
Layer 4 Sharing Controls — Recordings Cannot Go Viral

A malicious user's end goal after recording is to share the content. Sharing controls are the last line of defence and the most impactful for preventing revenge content distribution.

What Would Go Wrong
Unrestricted File Download and Sharing
If users can download the raw video file to their device gallery from within ARYAA, it immediately becomes shareable to any platform — WhatsApp, Telegram, Instagram, Twitter — with zero friction. Revenge content, private conversations recorded without consent, surveillance footage — all of it can be exported and distributed without any trace leading back to ARYAA or the original SOS event.
Controlled Sharing Channels Only
No Raw Download + Traceable Share Links
No direct download of raw video/audio files to the device from within ARYAA — ever. Sharing options are strictly limited to: (1) Generate a time-limited secure link that expires in 48 hours — for sharing with police or lawyers only. (2) Submit directly to a connected police complaint portal — evidence goes straight to authorities. (3) Share with your own emergency contacts only — no external sharing. Every share event is logged in the audit trail: who shared, what was shared, when, and to which recipient. Recipients of a shared link cannot forward it — links are tied to the recipient's phone number via OTP verification before playback. This means the chain of custody is always traceable and the recording cannot spread beyond the intended recipients.
Layer 5 Automatic Expiry — No Recording Library Through Fake SOS

Removing the incentive to accumulate recordings through repeated fake SOS triggers by ensuring recordings that are never acted on are automatically and cryptographically destroyed.

What Would Go Wrong
Surveillance Library Accumulation
Without expiry, a malicious user could trigger fake SOS events repeatedly — recording a target's private conversations over weeks or months. Each individual recording might not be enough for their purpose, but a library of 50 recordings gives them a comprehensive surveillance archive. Permanent storage makes ARYAA a long-term covert surveillance platform.
Tiered Automatic Deletion
Cryptographic Expiry by Use Case
Default retention: recording is automatically and cryptographically deleted after 30 days if the user has not explicitly saved it for legal purposes. Cryptographic deletion means the encryption key is destroyed — the file may still exist as ciphertext on S3 but is permanently unreadable. Saved recordings require the user to provide a reason (free text field — this creates a paper trail and a small deterrent). Maximum retention without an active legal case reference: 90 days. Recordings from cancelled SOS events where cancel happened within 10 seconds of trigger: deleted after 7 days — these are most likely accidental triggers or fake SOS. Recordings from cancelled-within-60-seconds SOS events: flagged for review before any access is allowed. The effect: someone building a surveillance library through fake SOS ends up with recordings that auto-delete every 30 days. Their malicious archive never accumulates.
Layer 6 Abuse Pattern Detection — Flagging Suspicious SOS Behaviour

Your internal analytics dashboard (from Recommended Additions) can identify patterns of SOS usage that are statistically inconsistent with genuine emergencies.

What Would Go Wrong
Repeated Fake SOS Below Individual Detection Threshold
A sophisticated malicious user spaces out their fake SOS events — one per week, always cancelled within 30 seconds, always in the same private location. No single event looks obviously suspicious. But the pattern over 8 weeks is clearly not a genuine emergency pattern. Without systematic pattern detection, this behaviour goes unnoticed until real damage has been done.
Statistical Pattern Flags + Human Review
Anomaly Detection on SOS Event Patterns
Flag for human review when: (1) User triggers SOS more than 5 times per week with no police involvement, no contact acknowledgement, and always cancels within 60 seconds. (2) SOS events always occur at the same private location that is not the user's registered home address. (3) SOS events occur on a regular schedule — every Tuesday at 7 PM for 4 consecutive weeks — inconsistent with random genuine emergencies. (4) SOS is triggered, immediately cancelled, and the device does not move at all during the SOS — suggesting it was triggered while stationary at a desk, not during a mobile emergency. Flags go to a human review queue — not automatic punishment. ARYAA team reviews the SOS event pattern and logs (never the recording content at this stage — privacy). If the pattern is clearly suspicious: account is temporarily suspended and user is asked to explain. The user can appeal with documentation of a genuine ongoing threat.
Legal Architecture Around Recording
📋 What Users Agree to at Onboarding
  • Recording is exclusively for personal safety emergency documentation
  • Using the recording feature outside genuine emergency situations violates Terms of Service and may violate Indian law
  • ARYAA maintains audit logs of all recording sessions and will provide these to law enforcement on valid legal request
  • Misuse of the recording feature may result in immediate account termination and referral to law enforcement
  • Recordings are stored encrypted and subject to automatic deletion per the retention policy
🔴 The Visible Recording Indicator

When recording is active, ARYAA displays a persistent red dot indicator on screen and a notification in the status bar — similar to how iOS shows an orange dot when the microphone is in use.

  • Indian courts have been more sympathetic to recording evidence where some disclosure existed — even if only visible to the person holding the phone
  • The visible indicator creates a psychological deterrent — a malicious user sees the red dot and knows a paper trail exists
  • Legally, ARYAA can argue the recording was not covert because an indicator was displayed
  • This does not require consent from the other party — it only requires that the recording's existence was not actively concealed by the platform
The Remaining Hard Cases — Being Honest About Limits
Hard Case 1 — Two People in a Mutual Dispute Both Trigger SOS

Two people in a genuine dispute both claim the other is a threat. Both trigger ARYAA SOS. Both record each other. Who is the victim? ARYAA's answer: You are not a court. Your job is to preserve evidence and provide it to law enforcement when legally requested. Both recordings go to police along with the complete SOS audit trails. Police determine what happened. Your dual audit trail — two simultaneous SOS events from the same location — is itself useful evidence of the confrontation's nature.

Hard Case 2 — Legitimate SOS Captures Third Party's Private Conversation

A user triggers SOS in a genuinely dangerous situation but the recording captures a third party's private conversation without that person knowing. ARYAA's answer: This is legally defensible as self-protection recording in India. The recording captures the user's immediate environment during a documented emergency — this is not surveillance, it is self-defence documentation. Your consent framework and Terms of Service make this distinction explicit. Indian courts have consistently distinguished between surveillance recording and self-protection recording.

Hard Case 3 — Sophisticated Bad Actor Uses a Second Phone as Their "Emergency Contact"

Someone sets up a second phone as their only emergency contact. They trigger fake SOS, their second phone acknowledges it (so no "no contact response" flag fires), and they use ARYAA purely as a covert recording tool with cloud backup. ARYAA's answer: The 24-hour access lock prevents immediate weaponisation. The 30-day auto-expiry prevents library accumulation. The sharing controls prevent easy distribution. The pattern detection flags repeated short-cancelled SOS events. The cost-benefit for this sophisticated setup versus a free screen recording app with cloud sync is so unfavourable that most malicious users will not pursue it. This is the irreducible minimum of misuse that no technical system can completely eliminate — but every layer above makes it substantially more difficult, more traceable, and less useful.

The honest answer to the misuse question: You cannot make misuse impossible. You can make it difficult, traceable, time-limited, and consequential. Every layer above adds friction and reduces the value of the recording to someone with malicious intent — while adding zero friction to someone in genuine danger. A determined bad actor who wants to covertly record someone has better tools available to them than a safety app that sends SOS alerts to their own contacts. ARYAA becomes a worse surveillance tool than a free screen recorder — and that is exactly the goal.

Recommended Additions

10 features not in the original plan that significantly improve safety, usability, and ARYAA's long-term potential

💡
These are not scope additions to Phase 1. They are identified improvements to be incorporated into the right phase at the right time. The priority table at the bottom maps each addition to effort, impact, and when it belongs.
Addition 1 — AI Predictive Safety Layer
Current Gap
ARYAA Is Purely Reactive
Every feature currently requires the user to trigger something. The app does nothing until you press a button. But in many dangerous situations — a car following you, a journey taking far too long, a deviation from your usual route — the danger is already developing before the user thinks to act. A purely reactive app misses the window where proactive intervention would have made the biggest difference.
Proposed Addition
Pattern-Based Predictive Alerts
ARYAA learns your patterns (with consent): your usual routes, typical journey times, regular check-in schedules. When you deviate — route change at night, journey 3× longer than usual, unusual stop in an isolated area — ARYAA quietly asks "Are you okay?" before anything bad happens. Voice tone analysis during phone calls can detect stress markers and offer a discreet alert option. Claude API can power the pattern analysis and anomaly detection logic. This turns ARYAA from a panic button into an actual safety companion.
Addition 2 — Automated Voice Calling Agent
Current Gap
SMS Is Read, Not Heard
SOS sends SMS. But emergency contacts may have their phone face-down, on silent, or in another room. An unread SMS during an emergency is the same as no alert. Currently if contact 1 doesn't respond, there is no automatic escalation — the user is hoping someone checks their messages. In a real emergency, hope is not a system.
Proposed Addition
Sequential Voice Call Escalation
If emergency contact 1 does not acknowledge the SOS within 30 seconds, an automated voice agent calls them: "This is an urgent safety alert from ARYAA. [Name] has triggered an emergency SOS at [location]. Please call them immediately or go to [address]. Press 1 to confirm you have received this message." If no response within 30 seconds — auto-escalate to contact 2, then 3. Built on Twilio Programmable Voice. Takes approximately 2 days to build. This is the difference between an alert being seen and an alert being acted on.
Addition 3 — Web Dashboard for Emergency Contacts
Current Gap
Contacts Need the App to Track
Emergency contacts currently receive an SMS with a tracking link that opens a basic webpage. This works for smartphone users. But parents in smaller towns, elderly relatives, or contacts using feature phones cannot effectively use an app or even a mobile-optimised tracking page. The contact who is most motivated to help — a parent — may be the least technically equipped to use the current system.
Proposed Addition
Real-Time Contact Web Dashboard
A full desktop-compatible web dashboard that opens in any browser — no app install, no account needed. Shows: live map with real-time location updates, SOS status and timeline, last 10 location history points, a prominent "I am responding — heading to this person now" button that notifies other contacts someone is already helping. Parents with no smartphone can open this on a desktop or laptop browser. Built as a React web app served from the same backend. Sharing via a one-click unique URL in the SOS SMS.
Addition 4 — Duress / Coercion Mode
Current Gap
SOS Can Be Forced Cancelled
If someone is physically coercing a user to unlock their phone and cancel the SOS — threatening violence if they don't — the current system has no defence. A visible, active SOS on screen is a liability when someone dangerous is watching. The user is forced to cancel, contacts stand down, and the real danger continues with no help coming. This scenario — forced cancellation under threat — is a realistic risk in exactly the situations ARYAA is designed for.
Proposed Addition
Duress PIN — Silent SOS Continuation
A secondary PIN the user sets up during onboarding. If entered during an active SOS "cancellation," it appears to cancel everything — SOS screen disappears, no visible indication — but silently continues the SOS in the background. Emergency contacts receive a special "DURESS" flag alert: "This person is under duress. SOS is active. Do not call back — contact police immediately." Modelled directly on bank ATM duress PIN systems. Architecturally: the SOS flow needs a "silent continue" state built in from Phase 1 even if the UI for it comes in Phase 2.
Addition 5 — What3Words Integration
Current Gap
GPS Coordinates Are Useless to Most People
An SOS SMS that says "Location: 18.5204° N, 73.8567° E" tells a panicking parent nothing. Even a Google Maps link requires smartphone access and data connectivity to open. In poorly mapped areas of Pune — construction zones, new layouts, inner lanes — the pin may not match any recognisable address. A police officer or ambulance driver needs a location they can act on immediately. The gap between "I know the coordinates" and "I can find this person" is where rescue time is lost.
Proposed Addition
What3Words Address in Every SOS SMS
What3Words divides the entire world into 3m × 3m squares, each with a unique 3-word address — for example "///table.lamp.river". This is more precise than a street address, works in areas without named roads, and is increasingly used by Indian emergency services. Add one extra line to every SOS SMS: "Precise location: ///word.word.word — open at w3w.co/word.word.word". What3Words has a free API tier (10,000 calls/month free). Cost to add: approximately 2 hours of development time. Impact: meaningfully faster rescue response, especially in low-infrastructure areas.
Addition 6 — Internal Analytics Dashboard
Current Gap
Zero Visibility Into How the App Is Used
Without analytics, you are building blind. You don't know which features users actually use, when SOS events spike, what percentage of SOS triggers are false alarms, which device models have background process failures, or where on the map most incidents are reported. This data determines what to build next, what to fix urgently, and what to show Pune Police when pitching a data partnership. Building without it is guesswork.
Proposed Addition
Metabase Dashboard on PostgreSQL
Metabase is a free, open-source analytics tool that connects directly to your PostgreSQL database and auto-generates dashboards with zero custom code. Set up in 2 days. Key dashboards to build: daily SOS triggers (real vs cancelled), feature usage funnel, SOS delivery latency (time from trigger to SMS delivered), geographic heatmap of incidents (anonymised), device model failure rates, peak usage hours. This data also becomes your pitch deck for Pune Police partnership ("Here is where incidents are clustering in Pune") and for investor conversations. Hosted inside your private VPC — never exposed publicly.
Addition 7 — Volunteer Responder Network
Current Gap
What If You Have No Trusted Contacts Nearby?
ARYAA's current SOS system assumes you have trusted contacts who are reachable and geographically close enough to help. But a student who just moved to Pune from another city, a solo traveller, or a night-shift worker whose family is in another state may have contacts who can do nothing except worry. Calling 112 is the only physical response option — and police response times in India average 15–30 minutes even in emergencies.
Proposed Addition
Opt-In Verified Volunteer Network
An opt-in network of verified ARYAA volunteers who agree to receive anonymous emergency alerts from strangers within 500 metres of them. When SOS fires: the 3 nearest verified volunteers get a discreet alert — "Someone nearby needs help. Are you able to assist? Their location: [map link]." Volunteers are verified via Aadhaar-linked identity check in partnership with an NGO. This is the feature that transforms ARYAA from an app into a movement — a community actively looking out for each other. Operationally complex (verification, liability, training) — Phase 3+ only. But it is the thing that earns national press coverage and government attention.
Addition 8 — Medical ID Card
Current Gap
First Responders Have No Medical Context
When an ambulance arrives after an ARYAA SOS, the paramedics know where the person is. They know nothing else. No blood type, no allergies, no medical conditions, no current medications, no doctor contact. For a diabetic user, a person with a severe allergy, or a senior citizen on blood thinners, this gap in information can be as dangerous as the emergency itself. Both iOS and Android have native Medical ID systems — but most users never fill them in.
Proposed Addition
ARYAA Medical ID — Lock Screen Accessible
A medical profile stored in ARYAA: blood group, allergies, chronic conditions, current medications, doctor name and number, emergency medical notes. Accessible directly from the phone lock screen — ARYAA registers itself as a lock screen widget/shortcut on both iOS and Android. ARYAA also writes this data to the native Medical ID on iOS (Health app) and Android's Emergency Info so any paramedic who knows to look finds it. Included in SOS SMS as a link: "Medical profile: [secure link]" — first responders can open it without any login. 2 weeks to build, enormous value for the senior citizen and medical condition personas.
Addition 9 — Dead Zone Check-In
Current Gap
ARYAA Is Blind in No-Signal Areas
If a user enters an area with no mobile signal — a remote trek, a basement parking lot, a rural road, a building with poor coverage — ARYAA has no way to know whether they are safe or in danger. They cannot trigger SOS (no signal). Their contacts see no location updates and don't know if the silence is normal or alarming. The offline SMS fallback helps, but only after the SOS is triggered — not when the user is completely unreachable.
Proposed Addition
Planned Dead Zone Mode with Timed Check-In
"I am going somewhere with no signal for the next [X] hours." User sets a dead zone timer before entering the area. ARYAA queues a check-in for when the timer expires. When connectivity returns: ARYAA prompts the user — "You've been unreachable for 2 hours. Are you safe? Tap to confirm." If no confirmation within 30 minutes of reconnecting: auto-alert contacts with last known location and note "User entered a no-signal area at [time] and has not checked in." Simple to build (2–3 days), covers a genuine gap for trekkers, remote workers, and anyone in poor-coverage areas.
Addition 10 — Post-Emergency Support Flow
Current Gap
ARYAA Has No "After" the Emergency
When a user cancels their SOS or marks themselves safe, ARYAA does nothing. The interaction ends. But the person may have just been through a frightening, traumatic, or violent experience. They may need to document what happened for legal purposes. They may be in emotional distress. They may want to report the incident. They may need to speak to someone. Right now ARYAA treats the cancelled SOS as the end of the story — for many users it is the beginning of a much harder chapter.
Proposed Addition
Post-SOS Recovery & Support Screen
After every SOS is cancelled or resolved, ARYAA shows a single gentle screen: "Are you okay? We're here." Three options: (1) "I need to talk to someone" — one-tap to iCall, Vandrevala Foundation, or iDiaries counselling. (2) "I want to document what happened" — private incident log: voice note, photos, timestamp, location — timestamped and encrypted, saved locally for potential legal use. (3) "I want to file a report" — links to Pune Police online complaint portal, cyber crime portal, or nearest police station map. Plus a 24-hour follow-up: a gentle push notification the next day — "How are you doing?" with the same three options. This is the recovery pillar of ARYAA's vision — prevent, respond, recover — that currently has zero features built for it.
